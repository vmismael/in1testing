## Start here — o que já existe para REUSAR
- **ffmpeg:** `lib/ffmpeg.ts` — `loadFFmpeg()` + `runConversion({file,outputName,args,mime,onProgress})`
  (single-thread; **nunca** core-mt nem COOP/COEP — quebra AdSense).
- **PDF:** `pdf-lib` (merge/split/rotate/draw) e `lib/pdfjs.ts` (`pdfjs`, worker pronto; v6:
  `render({canvas,viewport})`, `loadingTask.destroy()`).
- **Imagem/canvas:** `lib/image.ts` (`loadImage`/`drawToCanvas`/`canvasToBlob`).
- **UI/helpers:** `components/tools/file-dropzone.tsx`, `components/ui/copy-button.tsx`,
  `components/ui/progress.tsx`, `lib/download.ts` (`downloadBlob`/`stripExtension`),
  `lib/format.ts` (`formatBytes`).
- **Motor:** 1 objeto em `lib/tools/registry.ts` + 1 processador + `case` em
  `tool-processor.tsx` (`dynamic`, `ssr:false`) + 1 ícone no `iconMap`. Página/SEO/sitemap automáticos.

## Convenções / pegadinhas (já consolidadas)
- Base UI: `Button` usa `render`; `Slider.onValueChange` entrega `number | readonly number[]`;
  link estilizado = `buttonVariants` num `<Link>`.
- **eslint:** sem `setState` síncrono em `useEffect` (gerar em handlers / init lazy); evitar
  `performance.now()`/impuras no corpo do componente (extrair p/ helper de módulo).
- **Ícones:** confirmar o nome no lucide (`node -e "..."`) antes de adicionar ao `iconMap`.
- **SEO:** 5 `sections` com **≥800 palavras (só os `body`)**, 4 `howTo`, 5–6 `faq`, em inglês.
  Escrever ~165–175 palavras por section já de cara (evita reescrita).
- **WASM (vídeo):** guardas de tamanho + progresso + aviso de 1º uso (core cacheia depois).

## Passo 0 — Dependência única
- **Instalar:** `spark-md5` (+ `@types/spark-md5` se necessário) para MD5. SHA-1/256/512 via
  `crypto.subtle.digest` nativo. Tudo o mais já está instalado.

| # | Ferramenta | slug | Cat | Lógica |
|---|---|---|---|---|

|---|---|---|---|

## Sequência de execução
1. **Grupo A** (3 JS puro) — aquecimento, sem risco.
2. **Grupo B** (3 PDF) — reusa pdf-lib/pdf.js; `organize-pdf` é o de UI mais trabalhosa.
3. **Grupo C** (3 vídeo) — reusa `lib/ffmpeg.ts`; **verificar encoders** (libx264/libvpx/gif) num
   mini-teste antes da copy, como fizemos no spike de áudio.
4. `npm run build` + `npm run lint` após **cada** ferramenta; smoke test no navegador.

## Verificação (por ferramenta)
- Build gera `/<slug>` como SSG; lint limpo; sections ≥800; canonical/JSON-LD/sitemap ok.
- Browser: produz e copia/baixa o resultado; progresso nos tools de vídeo.
- **Anti-regressão:** nenhum header COOP/COEP (AdSense intacto).

## Decisões default já embutidas (ajustáveis)
- **hash-generator:** mostra MD5, SHA-1, SHA-256, SHA-512; cálculo ao vivo no handler.
- **color-converter:** aceita HEX (#rgb/#rrggbb), `rgb()` e `hsl()`; saída nos 3 + swatch.
- **rotate-pdf:** rotaciona **todas** as páginas pelo mesmo ângulo (por-página fica p/ backlog).
- **watermark-pdf:** marca d'água de **texto** (diagonal, opacidade ajustável); imagem fica p/ backlog.
- **trim-video:** `-c copy` por padrão (rápido; corte no keyframe mais próximo). Re-encode preciso
  fica como opção futura.
- **video-converter:** saídas **MP4 (H.264/AAC)** e **WebM (VP8/Vorbis)** — confirmar encoders no core.
- **video-to-gif:** default fps 10, largura 480px, **janela de tempo limitada** (ex.: até ~10s)
  e guarda de tamanho — GIF cresce rápido. Paleta (palettegen/paletteuse) opcional p/ qualidade.
- **Guardas de tamanho (vídeo):** ~150 MB, comunicadas na UI.

## Riscos / atenção
- **Encoding de vídeo é mais pesado que áudio** (Grupo C): libx264/libvpx existem no core padrão,
  mas confirmar + medir performance single-thread; GIFs longos podem estourar memória → limitar
  duração/largura.
- **organize-pdf**: UI de thumbnails + reordenar/excluir é a parte de maior esforço da onda.
- Mesma regra de ouro: **ffmpeg single-thread, sem COOP/COEP**.
